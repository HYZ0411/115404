## 第3章 系統架構

### 3-1 系統架構
分析出多數健身平台數據呈現過於線性且缺乏心理誘因，本系統旨在透過遊戲化八角框架與師徒制社群，將枯燥的訓練轉化為具備情感價值的數位資產。本系統採用 Serverless 無伺服器架構，整合 Flutter、Firebase 與 Vertex AI 技術棧，確保系統在具備快速落地上架能力的同時，亦達成商用級的數據安全性與運算效率。系統架構主要分為以下三個部分：

1. 前端介面與行動端整合：本系統採用 Flutter 跨平台框架開發，並透過 FlutterFire 插件深度串接 Firebase 官方服務。前端除負責呈現雷達圖、圓餅圖與折線圖等視覺化指標外，亦整合了 Firebase Cloud Messaging（FCM）推播機制。這確保了師徒社群間的訓練提醒與成就回饋能即時傳達。此外，系統導入 App Check 機制，從設備端限制只有合法的 App 實例能呼叫後端資源，有效防止 API 遭惡意盜刷。

2. 雲端後端服務與安全數據管理：採用 Firebase 作為核心雲端基礎設施。透過 Firebase Authentication 實作身分驗證，並利用 Cloud Firestore 達成訓練數據與社交資產的即時同步。為了保護機密金鑰（如 API Keys），系統採用 Cloud Functions 搭配 Secret Manager 作為後端邏輯中樞，將所有敏感運算與外部請求隔絕於雲端環境中。此設計確保了即便 App 遭反編譯，核心金鑰仍能維持絕對安全，符合 App Store 的資安規範。

3. 智慧導航與 AI 轉化引擎：本系統整合 Vertex AI in Firebase 實作正式環境的生成式 AI 功能。在開發階段透過 Google AI Studio 進行 Prompt 與回應品質的精準測試，確保 AI 建議的專業度。在執行階段，系統能根據 Firestore 中的即時紀錄，由 Cloud Functions 觸發 AI 進行雙階段輔助：訓練中的動態計畫調整以及訓練後的深度成長報告。此架構將隱性的汗水數據轉化為顯性的數位資產，透過 AI 的精準引導大幅提升使用者的心理酬賞與長期留存率。

### 3-2 系統軟、硬體需求與平台技術
本系統「Alchemy Workout 健金術」旨在打破健身初學者的「回饋沙漠」，因此在技術選型上，我們優先考量**高響應性、數據即時性與跨平台一致性**。本系統採用 Google 推出的 **Flutter** 作為跨平台開發框架，結合 **Firebase** 無伺服器（Serverless）架構與 **Vertex AI** 生成式人工智慧技術，構建出一個具備高擴展性且邏輯嚴密的智慧健身平台。

為了確保系統能精準實現「遊戲化八角框架」與「師徒社群」的複雜邏輯，我們在開發過程中導入了 **DDD（領域驅動設計）** 與 **BDD（行為驅動開發）**，將技術重心從單純的編碼轉向領域建模，並透過 AI 代理工具輔助實作，以達成商用級別的系統穩定度。以下針對本平台的核心技術優勢進行說明：

#### 表 3-2-1 「健金術」系統核心技術優勢
| 技術維度 | 說明 | 關鍵優勢 |
| :--- | :--- | :--- |
| **Flutter 跨平台框架** | 使用單一程式碼庫即可編譯為 Android 與 iOS 原生應用程式。 | 確保跨裝置介面美觀一致，大幅降低開發與維護成本。 |
| **Firebase 雲端生態** | 整合 Firestore 即時資料庫與雲端通訊（FCM）。 | 達成健身紀錄的秒級同步，支援師徒社群間的即時互動回饋。 |
| **Vertex AI 智慧引擎** | 串接 Gemini 系列模型，進行訓練計畫的動態調整。 | 消除使用者運動時的決策摩擦，將汗水數據轉化為成長報告。 |
| **Serverless 架構** | 透過 Cloud Functions 處理核心運算與 AI 邏輯。 | 提升系統安全性，確保 API 金鑰不外洩，並具備自動擴展能力。 |

考量到健身情境的高度移動性，本系統設計以行動端（Mobile App）為核心。為了讓使用者在各類網路環境（如健身房內部 Wi-Fi 或戶外行動網路）中均能流暢紀錄，我們針對硬體需求與軟體環境設定了明確的門檻。以下為系統運作之詳細需求表：

#### 表 3-2-2 系統硬硬體需求規格
| 系統需求項目 | 需求規格說明 |
| :--- | :--- |
| **裝置類型** | 支援 iOS 或 Android 系統之智慧型手機（主要目標裝置） |
| **作業系統版本** | Android 8.0 (API Level 26) 以上 / iOS 13.0 以上 |
| **處理器與記憶體** | 建議 4GB RAM 以上，以確保數據視覺化（雷達圖）流暢顯示 |
| **瀏覽器建議版本** | 若使用 Web 預覽版，建議使用最新版 Chrome、Safari 或 Edge |
| **網路連線需求** | 需具備 Wi-Fi / 4G / 5G 連線（支援 Firestore 離線數據快取） |
| **儲存空間需求** | 至少 200MB 可用空間（用於快取高效能訓練動作圖庫） |

透過上述軟、硬體技術的整合，「健金術」不僅能穩定承載高頻率的訓練數據，更能透過 AI 的介入，將冷冰冰的數據轉化為具備「情緒價值」的數位黃金，真正實踐遊戲化健身的願景。

### 3-3 使用標準與工具

本專題「Alchemy Workout 健金術」在開發流程上，不僅追求功能實現，更強調軟體工程的嚴謹度與開發效率的極大化。我們屏棄了傳統的勞動力密集型開發，轉而採用「AI 協作開發模式」，並嚴格遵循領域驅動設計（DDD）與行為驅動開發（BDD）標準。以下說明本系統開發過程中所運用的主要工具與技術規範：

* **開發框架與核心技術**
    * **Flutter & Dart**：Google 推出的跨平台 UI 工具箱。我們利用其高效能的渲染引擎實作複雜的「成長雷達圖」與「能力面板」，確保在 iOS 與 Android 平台上皆能提供流暢且一致的遊戲化視覺體驗。
    * **Firebase (Serverless)**：作為本系統的後端基石。利用 **Cloud Firestore** 達成健身數據的即時同步，並透過 **Cloud Functions** 處理後端邏輯與 AI 調用，減少維護實體伺服器的成本與風險。
    * **Vertex AI / Gemini API**：系統的智慧中樞。透過 Vertex AI 進行訓練計畫的動態決策與「數據黃金」的轉化運算，將枯燥的訓練紀錄提煉為具備行為指導價值的分析報告。
    * **Google AI Studio**：用於開發階段的 Prompt（提示詞）測試與優化環境。我們在此工具中精確調教 AI 教練的語氣與建議邏輯，確保其符合健身專業與「八角框架」的動機激勵。

* **軟體工程與 AI 開發工具**
    * **DDD (Domain-Driven Design)**：作為系統架構的最高指導標準。透過「事件風暴（Event Storming）」定義健身領域的邊界，確保系統邏輯（如：師徒機制、權重計算）能精準對應現實業務需求。
    * **BDD (Behavior-Driven Development)**：使用 Gherkin 語法撰寫規格書，確保開發者、AI 代理程式與使用者需求三者之間的高度對齊，降低邏輯摩擦。
    * **Claude Code**：作為本專案的「代理型 AI 開發助手」。我們將其定義為實作層的執行官，負責根據 BDD 規格自動化生成與重構代碼，使開發團隊能專注於高階的系統決策與架構設計。
    * **Visual Studio Code**：本組主要使用的程式碼編輯器，搭配 Flutter 與 Firebase 擴充套件，提供高效的開發、除錯與佈署環境。

* **設計、協作與專案管理**
    * **Figma**：專業的 UI/UX 設計工具。用於繪製「健金術」的高保真原型（High-Fidelity Prototype），並在開發前定義好所有視覺元素與遊戲化介面邏輯。
    * **Git / GitHub**：版本控制系統與協作平台。透過 GitHub 管理程式碼分支，並利用其 Issues 與 Pull Request 功能進行代碼審查與進度控管。
    * **Notion**：作為本專案的知識庫與任務管理中心。記錄 DDD 建模過程、會議記錄以及開發進度追蹤（Kanban），確保團隊成員資訊同步。
    * **Microsoft PowerPoint**：用於專題簡報與技術成果展示，將複雜的行為心理學機制視覺化。
