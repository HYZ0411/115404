# 專題系統手冊

## 第5章 需求模型

### 5-1 使用者需求

1. 功能性需求
*   帳戶與權限管理系統：
    *   第三方登入帳號：使用者能透過 Google 或 Apple ID 快速註冊並登入帳號。
    *   註冊個人資訊：使用者能完成個人資訊註冊，包含性別、身高、體重與年齡等基本資料。
    *   設定個人化資料：使用者能輸入健身目標、經驗、可運動時長及目前持有的器材。
    *   刪除帳號數據：使用者能依照隱私法規申請刪除帳號及其所有個人健身數據。
*   訓練與任務系統：
    *   完成新手教學：使用者進行簡單的新手教學，了解基本的APP使用方式。
    *   檢視健身計畫：使用者能檢視系統根據其個人化數據所推薦的健身計劃。
    *   執行訓練計劃：使用者能啟動並執行訓練計劃，記錄每個動作的組數、次數與重量等訓練數據。
    *   設定運動提醒：使用者能自主設定可運動的空閒時段，由系統在該時段發送提醒。
    *   接收訓練強度調整建議：系統（AI 智能教練）主動分析歷史紀錄後給予建議，動態微調訓練強度。
    *   接收替代動作建議： 使用者在訓練中遇到器材缺失或突發狀況時，系統（AI 智能教練）提供替代動作。
    *   監控訓練進度：使用者在訓練中能觀察剩餘組數、計畫完成百分比與進度條跳動。
    *   離線緩存健身紀錄：使用者在啟動 App 通過連網驗證後，可在健身紀錄模式下支援離線緩存，待網路穩定後自動同步數據。
    *   自定義訓練計畫：使用者能自由組合運動動作成訓練計劃。
    *   簽到累積紀錄：使用者能每日開啟 APP 進行簽到以累積連續紀錄。
*   遊戲化與社交系統：
    *   獲得隨機獎勵：使用者完成任務後，能隨機獲得金幣、抽獎券或補簽券。
    *   查看能力值面板：使用者能查看能力值（身體素質）隨訓練進度延伸的視覺化圖表。
    *   分享成果報告：使用者能生成具設計感的成果報告分享至社群平台。
    *   互動社群功能：使用者能為其他夥伴一鍵按讚、關注或發送社交刺激。
    *   參與分級排行：使用者能在排行榜中與同階級或實力相近的對手進行排名比拼。
    *   建立師徒關係：使用者可申請教官權限，並與新手建立師徒關係。
    *   展示成就獎杯：使用者能展示長期累積的稀有稱號、徽章與里程碑。
*   資產與商業系統：
    *   兌換虛擬資產：使用者能將累積的金幣兌換為補簽券、實體獎勵或虛擬特權。
    *   進行應用內購買：使用者能進行訂閱升級（如：付費解鎖進階功能）或儲值虛擬資產。
    *   查詢交易紀錄：使用者能查看過往購買訂閱、解鎖計畫或兌換獎項的歷史明細。

2. 非功能性需求
*   帳戶與權限管理系統：
    *   安全性（第三方登入、刪除帳號數據）：透過 Firebase Authentication 整合 Google / Apple OAuth 2.0 登入流程；身分驗證令牌（Token）由 Firebase 安全管理，不於本地明文儲存；導入 Firebase App Check 裝置合法性驗證，防止非法流量與 API 配額盜用；所有資料傳輸採 TLS 加密（Firebase 預設強制執行）。
    *   合規性（刪除帳號數據、註冊個人資訊）：符合 GDPR、CCPA 等個資法規；健康數據（身高、體重等）儲存於 Firestore 並設置嚴格讀寫規則（Security Rules），僅限本人存取；刪除請求需透過 Cloud Functions 執行級聯刪除，確保 Firestore、Authentication 及相關紀錄一併清除；需提供資料可攜性（Data Portability）。
    *   效能（第三方登入、設定個人化資料）：登入回應時間應在 2 秒內；個人資料寫入 Firestore 的 API 回應時間應在 1 秒內。
    *   可用性（第三方登入）：Google / Apple 登入服務中斷時，App 應顯示友善錯誤提示，不可崩潰；Firebase Authentication 離線狀態下需提示使用者重新連線，不得進入未驗證狀態。
*   訓練與任務系統：
    *   效能（接收 AI 訓練建議、推薦健身計畫）：AI 教練透過 Google AI Studio API 串接 Gemini 模型，結合 RAG 技術從 Firestore 動作資料庫動態擷取內容；端對端回應時間（含 Cloud Functions 調度與 Gemini 推理）應在 5 秒內完成；推薦計畫首次載入應在 3 秒內完成。
    *   效能（監控訓練進度、執行訓練計畫）：訓練中頁面狀態更新（組數、進度條）應在 100ms 內反映；計時功能精度誤差不超過 ±1 秒。
    *   可用性（離線緩存健身紀錄）：採前端本地緩存技術，確保在健身房弱網環境下仍可流暢記錄訓練數據；App 啟動時需連網完成 Firebase 身分驗證與 App Check 令牌初始化，驗證通過後方可進入離線模式；網路恢復後由系統自動同步至 Firestore，同步衝突以本地時間戳記為準；同步失敗需提示使用者並支援手動重試。
    *   安全性（離線緩存、執行訓練計畫）：本地緩存資料需加密儲存，防止裝置遺失造成數據外洩；App Check 令牌驗證機制確保每次啟動之合法性，防止偽造請求存取後端 API。
    *   易用性（完成新手教學、設定運動提醒）：新手教學步驟不超過 5 步，每步說明文字不超過 50 字；運動提醒設定流程應在 3 個操作步驟內完成；推播通知需遵守 iOS / Android 系統靜音與通知權限設定。
    *   可維護性（接收替代動作建議、接收訓練強度調整建議）：AI 調度邏輯集中於 Cloud Functions，可獨立版本化部署，不影響前端 Flutter 應用；Gemini 模型版本升級時，僅需更新 Cloud Functions 中的 API 串接設定；Firestore 動作資料庫的 schema 異動需向下相容，確保舊版 App 正常運作。
*   遊戲化與社交系統：
    *   可擴充性（參與分級排行、互動社群功能）：排行榜與社群動態資料儲存於 Firestore，利用其自動水平擴展能力應對流量成長；按讚、關注等寫入操作透過 Cloud Functions 處理核心邏輯，API 回應時間應在 1 秒內；社群動態採分頁查詢（每頁 20 筆），避免一次性讀取大量 Firestore 文件造成費用與效能問題。
    *   效能（查看能力值面板、分享成果報告）：能力值圖表資料從 Firestore 讀取後應在 2 秒內完成渲染；成果報告圖片由前端 Flutter 生成，產出時間應在 3 秒內，圖片大小不超過 2MB 以利分享至社群平台。
    *   安全性（建立師徒關係、互動社群功能）：教官申請資格審核邏輯集中於 Cloud Functions，前端無法繞過；師徒關係建立需雙方確認，相關狀態寫入 Firestore 並透過 Security Rules 限制篡改；社群內容需有檢舉機制，可觸發 Cloud Functions 進行自動審核。
    *   可用性（展示成就獎杯、獲得隨機獎勵）：獎勵發放邏輯集中於 Cloud Functions 並採冪等性設計（Idempotency），防止網路重傳導致重複發獎；成就與徽章資料持久化於 Firestore，確保長期累積的稀有獎章不因裝置更換而遺失。
*   資產與商業系統：
    *   安全性（進行應用內購買、查詢交易紀錄）：金流採用 Apple IAP / Google Play Billing 官方框架；購買憑據（Receipt）驗證邏輯集中於 Cloud Functions 進行伺服器端校驗，防止前端偽造；API 金鑰與敏感憑證透過 Google Cloud Secret Manager 管理，不寫入程式碼或版本控制。
    *   合規性（進行應用內購買、兌換虛擬資產）：需符合 App Store / Google Play 內購政策；虛擬貨幣（金幣）設計需避免觸發金融監管（如博彩法規）；退款政策需符合各地消費者保護法，並與 Apple / Google 退款機制整合。
    *   可用性（兌換虛擬資產、查詢交易紀錄）：兌換操作由 Cloud Functions 處理並採冪等性設計，防止重複扣點；金幣餘額透過 Firestore 即時同步，確保跨裝置顯示一致；交易紀錄查詢回應時間應在 2 秒內。
    *   效能（查詢交易紀錄）：歷史交易紀錄採分頁查詢（每頁 20 筆），利用 Firestore 游標（Cursor）分頁機制避免大量讀取；大量交易時需確保 Flutter 前端頁面不卡頓。

### 5-3 使用個案描述：使用活動圖描述

1. 使用個案名稱：第三方登入帳號
    *   行為者（Actor）：使用者
    *   前提（Precondition）：使用者已擁有 Google 或 Apple ID 帳號；App 已安裝並開啟
    *   結束狀態（Postcondition）：使用者成功登入健身 App，進入主畫面
    *   事件路徑（Event Flow）：
        *   Actor 動作：
            1. 使用者開啟 App，點擊「以 Google 登入」或「以 Apple ID 登入」
            3. 使用者在第三方頁面完成身份驗證
        *   系統回應：
            2. 系統跳轉至對應第三方授權頁面
            4. 系統接收授權 Token 並驗證合法性
            5. 系統建立或更新使用者帳號記錄
            6. 系統建立 Session，導向主畫面並顯示歡迎訊息
2. 使用個案名稱：註冊個人資訊
    *   行為者（Actor）：使用者
    *   前提（Precondition）：使用者已完成第三方登入；尚未填寫個人基本資料
    *   結束狀態（Postcondition）：個人基本資訊（性別、身高、體重、出生西元年）儲存至帳號
    *   事件路徑（Event Flow）：
        *   Actor 動作：
            1. 使用者進入個人資訊填寫頁面
            2. 使用者依序填寫性別、身高、體重與出生西元年
            3. 使用者點擊「儲存」
        *   系統回應：
            4. 系統驗證欄位格式與範圍是否合法
            5. 系統將資料寫入使用者個人檔案
            6. 系統顯示「資料儲存成功」並引導進入下一步驟
3. 使用個案名稱：設定個人化資料
    *   行為者（Actor）：使用者
    *   前提（Precondition）：使用者已完成個人資訊註冊
    *   結束狀態（Postcondition）：使用者健身目標、訓練經驗、可運動時長及持有器材資料儲存完成
    *   事件路徑（Event Flow）：
        *   Actor 動作：
            1. 使用者進入個人化設定頁面
            2. 使用者選擇健身目標
            3. 使用者填寫訓練經驗
            4. 使用者填寫每週可運動天數
            5. 使用者填寫每次可運動時長
            6. 使用者勾選目前持有的器材
            7. 使用者點擊「完成設定」
        *   系統回應：
            8. 系統驗證並儲存個人化偏好設定
            9. 系統根據設定推薦初始健身計畫，導向主畫面
4. 使用個案名稱：刪除帳號數據
    *   行為者（Actor）：使用者
    *   前提（Precondition）：使用者已登入帳號
    *   結束狀態（Postcondition）：使用者帳號及所有個人健身數據依隱私法規完整刪除
    *   事件路徑（Event Flow）：
        *   Actor 動作：
            1. 使用者進入「帳號設定」→「隱私與數據」
            2. 使用者點擊「申請刪除帳號」
            4. 使用者確認並輸入帳號驗證（密碼或第三方授權）
        *   系統回應：
            3. 系統顯示刪除範圍說明與不可逆警示
            5. 系統標記帳號為「待刪除」並啟動資料清除流程
            6. 系統於法規規定期限內完成刪除，並寄送確認通知
            7. 系統登出並導向未登入首頁
5. 使用個案名稱：完成新手教學
    *   行為者（Actor）：使用者
    *   前提（Precondition）：使用者首次登入 App 並完成個人化設定
    *   結束狀態（Postcondition）：使用者完成教學流程，App 核心功能解鎖
    *   事件路徑（Event Flow）：
        *   Actor 動作：
            3. 使用者依提示完成各步驟互動操作
            5. 使用者點擊「完成教學」
        *   系統回應：
            1. 系統偵測為新使用者，自動啟動新手教學流程
            2. 系統以逐步引導方式介紹主畫面、健身計畫、訓練記錄等核心功能
            4. 系統顯示每個步驟完成動畫與說明
            6. 系統標記教學完成，解鎖完整功能並顯示主畫面
6. 使用個案名稱：檢視健身計畫
    *   行為者（Actor）：使用者
    *   前提（Precondition）：使用者已完成個人化設定；系統已推薦健身計畫
    *   結束狀態（Postcondition）：使用者瀏覽並了解系統推薦的健身計畫內容
    *   事件路徑（Event Flow）：
        *   Actor 動作：
            1. 使用者進入「健身計畫」頁面
            3. 使用者點選計畫查看詳細內容
            5. 使用者可切換查看不同計畫
            6. 使用者選擇啟用某一計畫
        *   系統回應：
            2. 系統依使用者個人化資料顯示推薦計畫列表
            4. 系統顯示計畫名稱、主要肌群、目標、訓練動作及說明
            7. 系統設定為目前執行中計畫並更新主畫面
7. 使用個案名稱：自定義訓練計畫
    *   行為者（Actor）：使用者
    *   前提（Precondition）：使用者已登入；擁有建立自定義計畫的權限
    *   結束狀態（Postcondition）：自定義訓練計畫建立完成並可供執行
    *   事件路徑（Event Flow）：
        *   Actor 動作：
            1. 使用者進入「我的計畫」→「建立新計畫」
            2. 使用者輸入計畫名稱
            3. 使用者從動作庫搜尋並選擇訓練動作
            4. 使用者設定各動作的組數、次數、重量
            5. 使用者調整動作順序
            6. 使用者點擊「儲存計畫」
        *   系統回應：
            7. 系統驗證計畫內容並儲存至帳號
            8. 系統顯示計畫建立成功，新增至計畫列表
8. 使用個案名稱：設定運動提醒
    *   行為者（Actor）：使用者
    *   前提（Precondition）：使用者已登入；裝置已授予通知權限
    *   結束狀態（Postcondition）：指定時段的運動提醒設定完成，系統將於時段發送通知
    *   事件路徑（Event Flow）：
        *   Actor 動作：
            1. 使用者進入「設定」→「運動提醒」
            2. 使用者選擇欲提醒的星期與時段
            3. 使用者點擊「儲存提醒」
            6. 使用者點擊通知進入 App
        *   系統回應：
            4. 系統儲存提醒排程設定
            5. 系統於指定時段發送推播通知給使用者
            7. 系統導向當日訓練計畫頁面
9. 使用個案名稱：執行訓練計劃
    *   行為者（Actor）：使用者
    *   前提（Precondition）：使用者已選擇並啟用訓練計畫
    *   結束狀態（Postcondition）：訓練數據（組數、次數、重量）完整記錄至訓練歷史
    *   事件路徑（Event Flow）：
        *   Actor 動作：
            1. 使用者點擊「開始訓練」
